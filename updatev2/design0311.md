

# 深觅 AI Coach 产品设计报告

## 1. 产品概述

### 1.1 产品定位

#### 1.1.1 AI Coach 核心功能

深觅是一款面向个人成长与自我管理的AI Coach产品，其核心定位在于通过**长记忆AI助手技术**，为用户提供持续、个性化、深度理解的 coaching 体验。与传统的一次性对话AI不同，深觅强调**"记忆连续性"**——即AI能够跨越多次对话会话，持续积累对用户的认知，从而形成真正意义上的个人化辅助。产品名称"深觅"寓意"深度寻觅"，既指AI深入挖掘用户内在需求的过程，也象征用户通过产品实现自我探索与成长的旅程。

深觅的三大核心功能模块构成完整的用户成长闭环：**深潜**（对话与自我Coach）承担日常交互与信息输入职能，通过意图识别、分类记忆和角色化回复实现"自我coach"效果；**观己**（个人星图可视化）将分散的用户信息转化为可视化的知识图谱，提供自我认知的"上帝视角"；**追觅**（计划制定与自主执行）则基于前两者积累的智能，制定可执行的行动计划并通过IronClaw实现部分自主完成，将洞察转化为实际成果。这种"输入-认知-输出"的闭环设计，使得深觅区别于简单的问答工具，形成真正的 coaching 关系。

从技术实现角度，深觅的AI Coach功能建立在多层智能架构之上。最底层是**长记忆基础设施**，通过Mem0实现用户记忆的持久化存储与智能检索，支持情景记忆、语义记忆和程序记忆的多层抽象；中间层是**认知理解层**，利用LangChain和LangGraph构建复杂的对话处理流程，包括意图识别、信息提取、分类记忆和角色化回复；最上层是**交互呈现层**，通过精心设计的可视化界面，将AI的理解以直观、美观的方式呈现给用户。这种分层架构确保了系统的可扩展性和可维护性，同时为未来的功能迭代预留了充足空间。

#### 1.1.2 长记忆 AI 助手架构

深觅的长记忆AI助手架构是其区别于通用对话AI的核心技术差异化点。该架构采用**"双轨记忆+动态检索"**的设计范式，将用户相关信息划分为四个层次：**原始对话层**（Raw Conversations）、**提取事实层**（Extracted Facts）、**结构化记忆层**（Structured Memories）和**洞察推理层**（Inferred Insights）。这种分层设计借鉴了人类记忆的工作机制——感觉记忆、短时记忆、长时记忆的转化与巩固过程，使得AI能够模拟人类教练对用户背景的渐进式理解。

在具体实现上，**Mem0**作为核心记忆管理组件，负责记忆的存储、检索与更新。Mem0的独特价值在于其**"记忆即服务"**的抽象——不仅提供存储接口，更内置记忆提取、去重、摘要、关联等智能处理流程。当新对话产生时，Mem0自动识别值得保留的信息，与现有记忆进行语义去重，生成简洁摘要，并建立时序和主题关联。Mem0支持**基于相关性的智能检索**，当用户发起新对话时，系统不仅检索最近的对话，还会根据语义相似度检索历史上相关的记忆片段，这种检索机制通过**Chroma向量数据库**实现，将记忆内容嵌入为高维向量，支持高效的相似度搜索。

长记忆架构的另一个关键设计是**记忆的层次化组织与动态管理机制**。系统会自动识别记忆的重要性，对核心信息（如长期目标、关键偏好）进行高权重存储和频繁强化，对次要信息（如临时任务、一次性查询）则设置较低的保留优先级。每条记忆包含丰富的元数据：创建时间、最后访问时间、访问次数、来源对话、关联实体等，支持多维度的检索与关联分析。记忆的更新遵循**冲突解决规则**：同一主题的新记忆与旧记忆冲突时，根据时间戳、信息来源可靠性、用户明确确认等因素决定取舍，确保记忆库的一致性与时效性。

#### 1.1.3 三端适配策略（手机/网页/客户端）

深觅采用**"移动优先、渐进增强"**的三端适配策略，确保用户在手机、网页和客户端三个平台上获得一致且优化的体验。这一策略的核心是"一次开发，多端部署"，通过现代前端技术栈实现高效的跨平台开发，同时针对不同场景进行差异化优化。

| 平台      | 核心场景                | 设计重点            | 交互特性                      |
|:------- |:------------------- |:--------------- |:------------------------- |
| **手机端** | 碎片化即时交互（通勤、等待、睡前）   | 单手友好、快速进入、极简操作  | 底部固定导航、手势操作、语音输入优先、星图简化渲染 |
| **网页端** | 深度探索与规划（安静环境、大屏幕）   | 信息密度、复杂交互、效率工具  | 三栏布局、键盘快捷键、星图全屏沉浸、多窗口对比   |
| **客户端** | 重度用户持续陪伴（桌面常驻、系统集成） | 离线能力、系统级集成、性能优化 | 全局快捷键、推送通知、本地缓存、文件系统访问    |

**移动端适配**作为核心使用场景，承担日常对话、快速查看、即时通知等功能。设计遵循"拇指友好"原则：核心操作区域位于屏幕下半部分，重要按钮尺寸不小于44×44像素，支持手势操作（左滑删除、下拉刷新、长按菜单、双指缩放星图）。对话界面的输入框固定在底部，随键盘弹出自动调整位置。星图页面采用**可缩放画布+底部抽屉**的设计，用户可以通过双指缩放查看星图全貌，点击节点后从底部滑出详细信息面板，避免小屏幕上的信息拥挤。

**网页端适配**充分利用大屏幕的空间优势，提供更丰富的信息密度和更复杂的交互模式。对话页面采用**三栏布局**：左侧为历史会话列表，中间为主对话区域，右侧固定显示RAI（Recommended Action Items）小组件，实时展示意图识别结果和推荐行动。星图页面支持全屏沉浸模式，用户可以通过鼠标滚轮缩放、拖拽平移，右键点击节点唤出上下文菜单。网页端还支持键盘快捷键，如Ctrl+Enter快速发送、Esc关闭弹窗、/唤起命令面板等，提升高频用户的操作效率。

**客户端适配**（桌面应用）在网页端基础上增加了**系统级集成能力**：支持全局快捷键唤起悬浮对话窗口、系统通知推送、离线模式缓存、本地文件导入导出等。客户端采用Tauri或Electron框架开发，确保与网页端代码的高度复用，同时获得原生应用的性能和系统集成能力。三端数据通过统一的API服务实时同步，确保用户在任何设备上都能获得一致的认知连续性体验。

技术实现上，三端共享同一 **Vue.js 3 + TypeScript** 代码库，通过 **CSS Grid + Flexbox + 断点系统** 实现布局适配。断点设计遵循主流设备规格：移动端（<768px）、平板端（768-1024px）、桌面端（>1024px）。关键内容的容器最大宽度限制为1200px，超宽屏幕两侧留白，避免内容过度拉伸。状态管理采用 **Pinia**，配合 **localStorage/sessionStorage** 实现跨端数据同步，用户在不同设备上的对话历史和星图状态能够无缝衔接。

### 1.2 技术栈选型

#### 1.2.1 后端框架：FastAPI + SQLite

深觅的后端架构采用 **FastAPI** 作为主框架，这是一个基于Python 3.7+类型提示的高性能异步Web框架。FastAPI的选择基于多重考量：**性能层面**，FastAPI基于Starlette和Pydantic，利用Python的async/await语法实现高并发处理，基准测试显示其吞吐量接近NodeJS和Go框架，远超传统的Flask和Django；**开发效率层面**，FastAPI原生支持OpenAPI规范自动生成，类型提示驱动的代码补全和错误检测显著降低调试成本，前后端协作效率高；**生态兼容性层面**，FastAPI与AI/ML领域的Python库（LangChain、Hugging Face等）无缝集成，避免跨语言调用的复杂性。

数据库层选择 **SQLite** 作为MVP阶段的数据存储方案，这一决策体现了"简单优先、适度设计"的工程哲学。SQLite的**零配置、单文件存储**特性极大简化了部署流程，适合个人开发者和小规模验证场景。对于MVP阶段的单节点部署，SQLite的性能完全能够满足需求——其读写速度在单连接场景下可达每秒数万条记录，远超典型AI应用的QPS需求。更重要的是，SQLite与Python生态的集成极为顺畅，**SQLAlchemy ORM** 提供了完善的支持，开发效率高，且所有SQL操作通过抽象层实现，未来向PostgreSQL迁移只需修改连接配置，无需改动业务代码。

关键优化点在于连接池管理与 **WAL（Write-Ahead Logging）** 模式启用。通过 `aiosqlite` 实现异步访问，避免阻塞事件循环；WAL模式使读写操作可并发执行，缓解对话高峰期的写入压力。表设计遵循"宽表+JSON扩展字段"策略，如 `star_nodes` 表的 `metadata` 字段容纳动态属性，既保持schema稳定性，又支持业务快速迭代。

#### 1.2.2 AI 层：LiteLLM + Mem0 + LangChain + LangGraph

AI层是深觅的技术核心，采用**分层解耦、组合赋能**的设计理念，各组件分工明确又紧密协作。

| 组件            | 核心职责      | 关键特性                  | 深觅应用场景                            |
|:------------- |:--------- |:--------------------- |:--------------------------------- |
| **LiteLLM**   | 统一LLM调用网关 | 多模型路由、自动降级、成本追踪、流式输出  | 按任务复杂度动态选择模型，复杂推理→GPT-4，快速响应→轻量模型 |
| **Mem0**      | 长记忆管理     | 自动提取、智能检索、重要性评分、时间衰减  | 跨会话记忆保持、相关记忆唤醒、记忆冲突处理             |
| **LangChain** | LLM应用编排   | 提示词管理、工具调用、链式组合、RAG流程 | 对话流程模块化、外部工具集成、知识库检索              |
| **LangGraph** | 复杂工作流状态机  | 循环图结构、条件分支、并行执行、状态持久化 | 多轮对话管理、星图构建流程、计划执行监控              |

**LiteLLM** 作为模型路由层，提供统一的OpenAI兼容接口，支持100+种LLM后端，并内置重试、降级、负载均衡、成本追踪等企业级功能。这一设计使深觅能够**灵活切换底层模型**，根据场景需求在"能力-成本-延迟"三角中动态优化：复杂coaching对话调用GPT-4级模型，日常对话使用性价比更高的模型，网络不畅时自动降级到本地模型。

**Mem0** 承担长记忆管理的核心职责，其技术特性与深觅的产品需求高度契合。Mem0支持多种存储后端（Chroma/Qdrant/Pinecone用于向量存储，SQLite/PostgreSQL用于结构化存储），提供REST API和Python SDK两种集成方式，记忆提取过程由LLM自动完成无需人工规则。关键配置参数包括：embedding模型维度（默认1536维，与OpenAI text-embedding-3-small兼容）、向量相似度阈值（控制记忆检索的召回率与精确率平衡）、记忆重要性评分算法（基于信息熵和访问频次计算）。

**LangChain** 提供组件化抽象，深觅主要使用其Prompt模板管理、输出解析器、检索器、工具调用等能力。通过 **LCEL（LangChain Expression Language）** 构建可复用的处理管道，将意图识别、槽位填充、记忆检索、回复生成等环节模块化。

**LangGraph** 是工作流编排的核心，通过 **StateGraph** 定义节点和边，将对话处理建模为状态机。深觅的对话流程包含7个核心节点：输入接收→意图识别→记忆检索→角色选择→回复生成→记忆更新→星图同步。节点之间的边携带状态数据和转移条件，支持循环、条件分支、并行执行等高级特性。LangGraph的 **checkpoint机制** 保证了流程的可恢复性，即使服务重启也能从断点继续。

#### 1.2.3 向量存储：Chroma（RAG/GraphRAG）

**Chroma** 作为开源向量数据库，是深觅RAG和GraphRAG能力的基础设施。Chroma的核心优势在于：**轻量级部署**（支持内存模式和持久化模式）、**多语言SDK**（Python/JS/Rust）、**灵活的metadata过滤**、以及**内置的HNSW近似最近邻索引**。

Chroma的数据模型包含三个核心概念：**Collection（集合）**、**Document（文档）** 和 **Embedding（嵌入）**。深觅为不同用途创建独立的Collection：`memories_{user_id}` 按用户隔离确保隐私与性能，`documents_shared` 共享知识库，`star_nodes_{user_id}` 支持图遍历所需的批量相似性查询。每个向量记录包含完整的元数据信息，如user_id、timestamp、source_id等，支持精确的过滤查询。

| 检索模式         | 技术实现                               | 应用场景           | 性能特征             |
|:------------ |:---------------------------------- |:-------------- |:---------------- |
| **基础RAG**    | 查询向量化 → Chroma相似度搜索 → Top-K片段注入上下文 | 知识问答、资源推荐、概念解释 | 延迟<100ms，召回率>90% |
| **混合检索**     | 向量相似度 + metadata过滤（时间/类别/用户）       | 定向记忆检索、领域限定查询  | 精度提升15-20%       |
| **GraphRAG** | 实体识别 → 图谱遍历 → 子图向量精排               | 多跳推理、关联发现、路径分析 | 支持3-5跳关系查询       |

**RAG（Retrieval-Augmented Generation）** 流程的实现：用户查询经嵌入模型转化为向量，在Chroma中执行相似度搜索（默认余弦相似度，top-k=5），将检索到的文本片段与用户原始查询拼接，形成增强的prompt输入LLM。这一过程显著提升了回复的事实准确性和领域专业性。

**GraphRAG** 是RAG的进阶形态，通过构建结构化的知识图谱提升复杂查询的理解能力。与纯向量检索不同，GraphRAG首先识别查询中的实体和关系，在知识图谱中执行多跳推理（multi-hop reasoning），再综合图谱子结构和相关文本生成答案。例如，用户询问"我上次提到的那个考研复习计划和英语学习方法有什么关联"，GraphRAG能够定位"考研复习计划"和"英语学习方法"两个实体，遍历它们之间的关联路径（可能通过"雅思考试"节点连接），从而给出结构化的关联分析。

#### 1.2.4 自主执行：IronClaw 集成

**IronClaw** 是深觅实现"追觅"功能中自主任务执行的关键组件。IronClaw是一个AI智能体执行框架，设计目标是在**安全可控**的前提下，让AI能够调用外部工具、执行实际操作、完成复杂任务。与早期的Agent框架相比，IronClaw在安全性上做了关键改进：从直接操作文件系统转向数据库抽象层，通过WASM沙箱执行动态加载的工具代码，凭证加密存储且对LLM不可见，以及内置的提示词注入防御机制。

在深觅的集成架构中，IronClaw作为独立服务运行，通过HTTP API与主后端通信。当用户触发需要自主执行的计划项时（如"帮我搜索考研院校的报录比信息"、"预订一本缓解焦虑的心理学书籍"），主后端将任务描述和上下文封装为IronClaw任务，异步提交执行。IronClaw负责任务分解、工具选择、执行监控、结果汇总，最终将执行日志和结果回调给主后端。

| 任务类型     | 典型场景             | 工具集                     | 安全级别       |
|:-------- |:---------------- |:----------------------- |:---------- |
| **信息检索** | 学校排名查询、课程比价、新闻监控 | 搜索引擎、学术数据库、电商API        | 自动执行，无需确认  |
| **内容生成** | 邮件草稿、报告摘要、代码片段   | LLM生成、模板引擎              | 自动执行，可选预览  |
| **工具操作** | 日历预约、提醒设置、待办同步   | Google/Outlook API、系统通知 | 用户确认后执行    |
| **数据分析** | 情绪趋势、学习进度、效率统计   | Pandas、可视化库             | 自动执行，结果可审计 |
| **交易操作** | 课程购买、预约付费        | 支付网关、预订系统               | **强制人工确认** |

IronClaw的任务类型规划涵盖信息检索类、内容分析类、操作执行类、以及情感支持类。每类任务配置相应的工具集和执行策略，**关键操作设置人工确认节点**，平衡自动化效率与风险控制。执行状态通过 **Webhook回调 + 轮询查询** 双机制追踪，用户界面实时展示进度，支持中途取消和异常处理。

### 1.3 视觉设计规范

#### 1.3.1 主色调：蓝色系（#1E3A5F / #4A90D9 / #87CEEB）

深觅的视觉识别系统以**蓝色系**为核心，构建专业、信任、宁静的品牌感知。蓝色在色彩心理学中与理性、可靠、冷静相关联，非常契合AI Coach产品需要建立的用户信任关系；同时，蓝色也是科技产品的经典选择，能够传递"智能、未来、可靠"的品牌感知。

| 色彩层级     | 色值      | 名称         | 应用场景                 | 心理暗示     |
|:-------- |:------- |:---------- |:-------------------- |:-------- |
| **深海蓝**  | #1E3A5F | Deep Sea   | 主标题、关键按钮、品牌标识、导航栏    | 权威、深度、专注 |
| **天际蓝**  | #4A90D9 | Sky        | 交互元素、链接、高亮状态、AI消息气泡  | 开放、信任、行动 |
| **浅蓝**   | #87CEEB | Light Blue | 背景装饰、次要信息、hover状态、点缀 | 轻松、清晰、友好 |
| **极浅灰蓝** | #F0F4F8 | Pale Blue  | 页面背景、卡片底色、输入框背景      | 干净、舒适、现代 |
| **深蓝黑**  | #1A1A2E | Deep Navy  | 正文文字、重要信息、图标         | 稳重、易读、专业 |

色彩的使用遵循 **"60-30-10"原则**：60%的极浅灰蓝作为背景主导，30%的白色/浅灰用于内容区域，10%的蓝色系用于关键交互元素。这种配比确保了视觉的舒适性和层次感的清晰。**深色模式**作为可选项，将背景反转为#0F1419，文字相应调整为浅灰色系，蓝色主色调保持不变以确保品牌识别的一致性。

#### 1.3.2 简洁风格原则

深觅的界面设计遵循 **"少即是多"的极简主义原则**，核心目标是降低用户的认知负荷，让注意力聚焦于内容与交互本身。具体的设计准则包括：

**信息密度控制**：单屏展示的核心信息不超过**7±2个组块**（符合人类工作记忆容量），次要信息通过折叠、展开、层级深入等方式渐进披露。对话界面中，AI回复优先展示核心结论，详细推理过程可点击"查看思考过程"展开；星图默认只展示L1-L2层级节点，深层节点按需加载。

**视觉噪音消除**：去除不必要的装饰元素（阴影、渐变、边框），采用扁平化设计；图标使用线性风格，保持一致的2px描边宽度；动画效果克制，仅用于状态过渡和反馈提示，时长控制在**200-300ms**。

**排版呼吸感**：generous的留白（padding/margin），正文行高**1.6-1.8**，段落间距大于行间距，确保阅读舒适。中文内容优先使用系统字体（PingFang SC / Microsoft YaHei），减少字体加载开销。

**一致性约束**：建立**设计令牌（Design Tokens）**系统，所有颜色、间距、圆角、阴影值集中管理，确保跨组件、跨页面、跨平台的一致性。任何新组件的设计必须基于现有令牌组合，禁止随意新增样式值。

#### 1.3.3 响应式布局标准

三端适配的实现基于统一的**响应式网格系统**。移动端采用4列网格，断点宽度375px（iPhone SE）至428px（iPhone 14 Pro Max）；平板端采用8列网格，断点768px至1024px；桌面端采用12列网格，断点1280px至1920px及以上。关键内容的容器最大宽度限制为**1200px**，超宽屏幕两侧留白，避免内容过度拉伸。

| 断点     | 宽度范围        | 网格列数 | 典型布局           | 关键适配                |
|:------ |:----------- |:---- |:-------------- |:------------------- |
| **xs** | < 576px     | 4列   | 单列堆叠，底部固定导航    | 触摸目标≥44px，手势优先，语音输入 |
| **sm** | 576-768px   | 8列   | 可展开侧边抽屉        | 简化星图交互，模态层替代多窗口     |
| **md** | 768-1024px  | 8列   | 双栏布局（导航+内容）    | 侧边栏可折叠，星图支持拖拽       |
| **lg** | 1024-1440px | 12列  | 三栏布局（导航+内容+辅助） | RAI小组件常驻，键盘快捷键      |
| **xl** | > 1440px    | 12列  | 最大内容区1200px居中  | 多窗口模式，星图全屏沉浸        |

组件的响应式行为分为三类：**流动型**（宽度随容器自适应，如输入框、卡片）、**断点切换型**（在不同断点改变布局结构，如导航栏从底部标签切换为侧边栏）、**隐藏型**（在小屏幕隐藏次要功能，如星图的详情面板改为全屏浮层）。图片和媒体资源使用srcset提供多分辨率版本，确保各设备的显示清晰度与加载性能平衡。

触摸交互的适配：移动端所有可点击元素的最小尺寸为**44×44px**（Apple HIG标准），支持滑动手势（星图的缩放/平移、对话列表的左滑删除）；桌面端支持悬停状态预览、右键上下文菜单、键盘快捷键等精细交互。所有交互状态（default/hover/active/disabled/loading）均有明确的视觉反馈，确保用户的操作意图得到即时确认。

## 2. 核心功能模块设计

### 2.1 深潜（对话与自我 Coach）

#### 2.1.1 功能定义

##### 2.1.1.1 多轮对话引擎

深潜模块的多轮对话引擎是用户与AI Coach建立深度连接的核心通道。与单次问答不同，多轮对话要求系统维护完整的对话历史，理解上下文依赖，处理话题切换与回归，并在长对话中保持连贯性和一致性。引擎采用 **LangGraph构建状态机工作流**，每个对话会话对应一个StateGraph实例，状态对象包含：messages（历史消息列表）、current_intent（当前意图）、extracted_slots（已提取的槽位）、retrieved_memories（检索到的相关记忆）、coach_role（当前匹配的角色策略）等字段。

对话引擎支持三种交互模式：**自由对话**（用户主导话题方向）、**引导对话**（AI基于目标主动引导）、**结构化对话**（针对特定场景的固定流程，如目标设定、情绪评估）。模式切换由意图识别节点动态判断：当检测到用户有明确需求但信息不完整时，自动进入引导模式；当用户表达模糊或情绪化时，采用自由模式建立信任；当识别到标准化场景（如"我想设定一个考研目标"）时，触发结构化流程。

##### 2.1.1.2 意图识别系统

意图识别是深潜模块的智能中枢，负责将用户的自然语言输入映射为预定义的意图类别，并提取关键参数（槽位）。系统采用 **"规则+模型"的混合架构**：轻量级规则（正则表达式、关键词匹配）处理高频、明确的意图，确保响应速度；LLM-based分类器处理复杂、模糊的表达，确保覆盖广度。

| 意图大类     | 子类别             | 示例表达                     | 处理策略        |
|:-------- |:--------------- |:------------------------ |:----------- |
| **自我探索** | 情绪表达、价值观澄清、优势识别 | "我最近很焦虑"、"我不知道自己擅长什么"    | 共情回应+开放式提问  |
| **目标管理** | 目标设定、进度追踪、障碍讨论  | "我想三个月减重10斤"、"考研复习进度落后了" | GROW模型引导    |
| **知识获取** | 概念解释、方法建议、资源推荐  | "什么是正念冥想"、"如何提高专注力"      | RAG检索+结构化输出 |
| **行动支持** | 计划制定、任务分解、执行提醒  | "帮我安排下周的学习计划"            | 触发追觅模块      |
| **关系互动** | 社交建议、冲突处理、沟通技巧  | "怎么跟导师沟通研究方向"            | 情景模拟+角色扮演   |
| **系统操作** | 查看星图、修改记忆、功能导航  | "打开我的星图"、"忘掉我刚才说的"       | 直接执行或确认     |

意图识别的**置信度分数**（confidence_score）决定后续处理路径：高于0.8直接执行对应策略，0.5-0.8触发澄清确认（"你是想...还是..."），低于0.5进入通用对话模式并记录用于模型优化。槽位提取采用与意图识别联动的方案，在识别意图的同时，从文本中提取时间、地点、人物、数值等实体，填充到预定义的槽位结构中。

##### 2.1.1.3 信息提取与槽位填充

信息提取是将非结构化的用户输入转化为结构化数据的关键步骤，为后续的记忆存储和星图构建提供标准化输入。深潜模块的槽位体系涵盖四大类二十余个槽位：

| 槽位类别     | 具体槽位                                                                             | 取值类型      | 验证规则           |
|:-------- |:-------------------------------------------------------------------------------- |:--------- |:-------------- |
| **基础信息** | user_name, age_range, location, occupation, life_stage                           | 字符串/枚举    | 格式合法性检查        |
| **目标相关** | goal_description, goal_category, target_date, success_criteria, current_progress | 字符串/日期/数值 | 日期不能为过去，评分1-10 |
| **偏好相关** | learning_style, communication_tone, feedback_frequency, preferred_time           | 枚举        | 预定义选项集         |
| **状态相关** | current_mood, stress_level, energy_level, recent_events                          | 字符串/数值/日期 | 评分范围校验，时间合理性   |

槽位填充采用 **"提取-验证-确认"的三阶段流程**。**提取阶段**由LLM从用户输入中识别候选值；**验证阶段**检查候选值的格式合法性和语义合理性；**确认阶段**将提取结果以自然语言反馈给用户（"我记录一下，你的目标是在2025年6月前通过雅思考试，对吗？"），用户确认或修正后才正式写入记忆系统。这种设计既保证了数据的准确性，也增强了用户的控制感和信任度。

##### 2.1.1.4 分类记忆机制

分类记忆机制是深潜模块与Mem0长记忆系统的核心交互接口。每次对话结束后，系统自动分析对话内容，识别值得长期保存的信息，按照记忆类型分类存储。Mem0的四类记忆模型在此得到具体应用：

| 记忆类型     | 定义         | 提取触发                     | 持久性策略                      |
|:-------- |:---------- |:------------------------ |:-------------------------- |
| **工作记忆** | 当前会话的短期上下文 | 每轮对话自动维护                 | 会话结束后丢弃                    |
| **事实记忆** | 用户的稳定属性信息  | 明确陈述个人信息、多次推断的 stable 模式 | 最高优先级，importance_score≥0.8 |
| **情景记忆** | 具体的对话事件记录  | 重要事件、关键决策、情感高峰           | 30天后摘要压缩，原始记录归档            |
| **语义记忆** | 抽象的行为模式和洞察 | AI定期分析历史数据，用户确认后纳入       | 跨场景迁移应用，持续强化               |

**工作记忆**仅存在于单次会话内，会话结束后丢弃，用于维护即时上下文，在LangGraph的状态对象中实现。**事实记忆**存储用户的稳定属性，如"用户名叫李明"、"用户是素食主义者"，具有最高的持久性优先级。**情景记忆**记录具体的对话事件，如"2025年3月5日，用户提到考研复习压力大"，支持时间范围查询和主题检索。**语义记忆**则从多次交互中抽象出行为模式和通用知识，如"用户倾向于在晚间处理重要决策"，实现跨场景的迁移应用。

记忆的检索策略同样分类处理：对话开始时，优先检索与当前话题相关的目标记忆和近期情绪记忆，建立上下文；对话进行中，根据意图识别结果动态检索相关类别；生成建议时，综合检索所有类别，确保建议的个性化和一致性。Mem0的向量相似度检索为这种分类检索提供了基础能力，而深觅在上层增加了分类过滤和优先级排序逻辑。

##### 2.1.1.5 角色化回复策略

角色化回复策略使深觅能够根据场景和用户状态，动态调整AI的回应风格和内容重点。系统预定义多种Coach角色，每种角色有明确的定位、语气特征、常用技巧和禁忌事项：

| 角色名称               | 定位        | 语气特征                 | 适用场景        | 禁用行为        |
|:------------------ |:--------- |:-------------------- |:----------- |:----------- |
| **陪伴者（Companion）** | 情感支持、倾听理解 | 温暖、共情、非评判            | 用户情绪表达、压力倾诉 | 急于给建议、打断倾诉  |
| **探询者（Explorer）**  | 启发思考、拓展视角 | 好奇、开放、提问为主           | 自我探索、价值观澄清  | 直接下结论、封闭性问题 |
| **导师（Mentor）**     | 经验分享、方法指导 | 专业、结构化、有案例           | 知识获取、技能学习   | 空洞说教、脱离上下文  |
| **教练（Coach）**      | 目标导向、行动推动 | 直接、挑战、accountability | 目标设定、进度追踪   | 过度施压、忽视情绪   |
| **伙伴（Partner）**    | 平等协作、共同创造 | 轻松、幽默、互动感            | 创意发散、计划制定   | 单向输出、缺乏互动   |

角色选择由意图识别节点结合用户状态综合判断：情绪指标（如压力水平>7）优先触发陪伴者角色；明确的目标相关意图触发教练角色；开放式探索类意图触发探询者角色。角色选择结果记录在状态对象中，影响后续prompt的系统消息（system message）构建。用户也可主动指定角色（"今天我想让你当我的教练"），系统尊重用户选择并记录偏好。

#### 2.1.2 前端界面设计

##### 2.1.2.1 对话气泡布局（用户/AI 区分）

对话界面的气泡设计遵循"清晰区分、自然流动"的原则。用户消息采用**右对齐布局**，气泡背景使用天际蓝（#4A90D9）的浅色变体（#E8F4FD），文字为深蓝黑（#1A1A2E），形成足够的对比度确保可读性。AI消息**左对齐**，气泡背景为白色，左侧搭配AI头像（深觅品牌图标，圆形，40px），增强人格化感知。

气泡的形状设计：用户消息气泡右上角为尖角（指向用户输入区），其他角圆角12px；AI消息气泡左上角为尖角，其他角圆角12px。这种不对称设计在视觉上暗示了对话的方向性。气泡的最大宽度为容器宽度的75%（移动端）或60%（桌面端），超长的内容自动换行，避免单条消息占据过多垂直空间。

时间戳显示策略：同一分钟内连续的消息只显示最后一条的时间；超过5分钟的消息间隔显示时间分隔线（"14:32"）；日期变更时显示日期分隔线（"昨天"、"3月10日"）。这种渐进披露避免信息过载，同时保留必要的时间上下文。

##### 2.1.2.2 实时意图识别可视化组件（RAI Badge）

实时意图识别可视化（Real-time AI Badge, **RAI Badge**）是深潜界面的创新设计，旨在增强系统的透明度和可解释性。当用户输入消息并发送后，在AI回复生成前，界面在输入框上方显示动态的处理状态条，以微动画展示当前处理阶段："理解中..."（意图识别）→"回忆中..."（记忆检索）→"思考中..."（回复生成）。

处理完成后，RAI Badge以标签形式展示识别到的意图类别（如"目标设定"、"情绪支持"），用户点击标签可展开查看详细信息：提取的槽位键值对、检索到的相关记忆摘要、当前激活的Coach角色。这种设计满足了用户的知情权，建立了对AI决策过程的理解和信任，也为高级用户提供了干预入口（如发现意图识别错误可手动修正）。

RAI Badge的视觉风格与整体设计语言一致：标签使用浅蓝色背景、深蓝色文字，圆角16px（pill形状），图标+文字的组合形式。展开后的详情面板采用卡片式设计，信息分组清晰，支持一键复制原始数据用于反馈。

##### 2.1.2.3 输入框与快捷操作栏

输入区域是用户与AI交互的主要入口，设计追求"零摩擦"体验。输入框固定在屏幕底部，高度自适应（最小56px，最大200px），支持多行文本输入和自动换行。占位符文案根据上下文动态变化：新会话显示"告诉我你的想法..."，对话中显示"继续聊..."，检测到用户犹豫（输入停顿超过3秒）显示"或者，你可以问我..."。

快捷操作栏位于输入框上方，高度44px，滚动展示预设的快捷操作。快捷操作的来源包括：AI基于当前上下文推荐的下一步（如用户提到压力大时推荐"试试呼吸练习"）、用户高频使用的自定义快捷语、以及系统功能入口（"查看星图"、"制定计划"）。快捷操作以chip形式呈现，点击后自动填充输入框或直接发送，减少用户的输入负担。

输入框右侧的功能按钮：发送按钮（纸飞机图标，用户输入非空时高亮）、语音输入按钮（麦克风图标，长按录音）、附件按钮（加号图标，支持图片、文件、位置分享）。语音输入支持实时转文字，转写过程中显示波形动画，确认后发送文字内容。

##### 2.1.2.4 历史对话时间轴

历史对话时间轴以侧边栏形式呈现（桌面端）或从底部滑出的抽屉形式（移动端），支持用户快速浏览和跳转过往会话。时间轴采用"日期分组+会话卡片"的层级结构：顶部显示"今天"、"昨天"、"本周"、"更早"的分组标题，每组内按时间倒序排列会话卡片。

会话卡片展示关键信息：会话标题（AI自动生成，如"关于考研规划的讨论"）、最后消息摘要（前50字符）、消息数量、最后更新时间。卡片hover状态显示操作按钮：继续对话、重命名、删除。点击卡片主区域加载对应会话，对话内容平滑滚动到顶部。

搜索功能集成在时间轴顶部，支持按关键词搜索会话标题和内容，搜索结果高亮匹配文本。筛选器支持按时间范围、话题标签、消息数量等维度过滤，帮助用户在大量历史记录中快速定位。

#### 2.1.3 交互流程

##### 2.1.3.1 用户输入 → 意图识别 → 记忆检索 → 角色匹配 → 生成回复

深潜模块的核心交互流程由LangGraph状态机严格编排，确保每个处理阶段的输出质量，并为调试和优化提供清晰的观测点。流程的详细步骤如下：

| 阶段  | 节点名称     | 核心操作              | 输出                                           | 延迟目标       |
|:--- |:-------- |:----------------- |:-------------------------------------------- |:---------- |
| 1   | 输入接收与预处理 | 文本清洗、类型检测、非文本预处理  | 清洗后的文本、输入元数据                                 | <50ms      |
| 2   | 意图识别     | 分类模型调用，槽位提取       | intent_category, confidence, extracted_slots | <200ms     |
| 3   | 记忆检索     | 向量语义搜索+结构化过滤      | retrieved_memories（Top-5，含相关度分数）             | <300ms     |
| 4   | 角色匹配     | 多因素加权决策           | selected_role, role_config                   | <50ms      |
| 5   | 上下文组装    | 整合系统提示、记忆、历史、当前输入 | 完整LLM输入上下文                                   | —          |
| 6   | 回复生成     | LiteLLM流式调用       | 生成的回复文本、token消耗                              | 2-3s（复杂查询） |
| 7   | 后处理与交付   | 格式检查、引用标记、响应构建    | 响应对象，异步触发记忆/星图更新                             | <100ms     |

**阶段1：输入接收与预处理**（Input Node）。接收用户原始输入，执行基础清洗（去除首尾空白、规范化标点），检测输入类型（文本/语音/图片/文件），非文本输入触发相应的预处理流程（语音转文字、图片OCR、文件解析）。

**阶段2：意图识别**（Intent Node）。调用意图分类模型，输出意图类别、置信度分数、提取的候选槽位。如果置信度低于阈值，触发澄清子流程；如果检测到多意图，按优先级排序并确认执行顺序。

**阶段3：记忆检索**（Memory Node）。基于当前输入和已识别意图，构建检索查询。检索策略包括：关键词精确匹配（用户名称、特定目标）、向量语义搜索（相关话题的历史讨论）、结构化过滤（特定时间范围、记忆类型）。检索结果按相关性排序，取top-5注入上下文。

**阶段4：角色匹配**（Role Node）。根据意图类别、用户当前状态（情绪、精力）、历史偏好，从角色库中选择最匹配的Coach角色。角色选择考虑多因素加权：意图-角色映射表（预设权重0.4）、用户显式偏好（权重0.3）、当前情绪状态（高压力优先陪伴者，权重0.2）、历史效果反馈（该角色过往的用户满意度，权重0.1）。

**阶段5：上下文组装**（Context Node）。整合所有前置信息，构建完整的LLM输入：系统消息（角色定义+全局约束）、relevant memories（格式化后的相关记忆）、conversation history（截断至上下文窗口允许的最大长度）、current user input。对于长对话，采用"摘要+近期详细"的层次化表示，确保关键历史不丢失。

**阶段6：回复生成**（Generate Node）。调用LiteLLM统一的模型接口，传入组装好的上下文，设置生成参数（temperature根据角色调整：陪伴者0.7更温暖，教练0.3更聚焦）。支持流式输出，token级实时渲染到界面。

**阶段7：后处理与交付**（Output Node）。对生成内容执行格式检查（避免敏感信息、确保JSON可解析如果要求结构化输出）、添加引用标记（如果使用了RAG检索的内容）、构建响应对象。同步触发记忆更新和星图更新的异步任务，不阻塞响应返回。

##### 2.1.3.2 关键信息自动提取与确认

关键信息的自动提取是确保记忆质量的关键环节，采用 **"提取-候选-确认"的交互模式**，平衡自动化效率与用户控制感。提取触发条件包括：检测到明确的个人信息陈述（"我叫..."）、目标相关表达（"我想..."）、偏好表达（"我喜欢..."）、重要事件（"我下周要..."）。

提取算法基于LLM的NER（命名实体识别）能力，结合领域特定的规则增强。对于高置信度的提取（如明确的姓名、日期），系统以微妙的方式确认（"好的，我记下了"）；对于中置信度的提取（如推断出的偏好），以建议形式呈现（"我注意到你经常晚上使用深觅，需要我把你的活跃时段设为晚间吗？"）；对于低置信度或敏感信息（如健康相关），必须获得明确确认后才存储。

确认交互的设计：非侵入式，以inline形式出现在AI回复的末尾或作为可点击的chip；支持一键确认（"是的"）和修正（"不对，是..."）；用户选择"忽略"则该信息不进入记忆系统，但记录忽略行为用于优化提取算法。所有确认操作反馈到记忆系统，更新提取规则的准确性评分。

##### 2.1.3.3 记忆强化反馈机制

记忆强化反馈机制是深觅持续学习、越用越懂用户的核心设计。机制包含三个层面：

**即时反馈层**：每次对话结束后，界面显示"这次对话有帮助吗？"的评分请求（👍/👎），以及可选的文字反馈。评分数据关联到该次对话使用的记忆片段，用于评估记忆检索的相关性质量。

**周期性回顾层**：每周生成"记忆周报"，向用户展示本周新增的关键记忆、记忆的使用情况（哪些被频繁检索、哪些从未被使用）、以及AI基于记忆生成的洞察（"这周你3次提到工作压力，需要聊聊吗？"）。用户可在此界面编辑、删除、合并记忆条目，修正AI的理解偏差。

**主动优化层**：系统每月执行记忆质量分析，识别低质量记忆（从未被检索、用户反馈负面、与其他记忆矛盾）并归档；识别高价值记忆模式（频繁使用、用户确认、产生积极结果）并强化；基于记忆网络分析发现潜在关联（"你提到想学英语，同时关注了几个留学账号，是在准备出国吗？"），主动发起探索性对话。

### 2.2 观己（个人星图可视化）

#### 2.2.1 功能定义

##### 2.2.1.1 用户信息全景图谱

"观己"模块的个人星图是深觅最具创新性的功能，它将AI对用户的所有理解——从离散的记忆片段、提取的事实信息、识别的行为模式——整合为一个**动态、可视化的知识图谱**。星图的核心价值在于"整体大于部分之和"：单独的每条记忆是碎片，而星图揭示了碎片之间的关联，帮助用户看到自己未曾察觉的模式和联系。

星图的构建数据源包括：Mem0中的四类记忆（事实、情景、语义、工作记忆）、对话中提取的槽位信息、追觅模块生成的计划和执行记录、以及用户主动添加的个人信息。这些数据经过**实体识别、关系抽取、消歧对齐**等处理，映射为图结构中的节点和边。节点代表实体（用户、目标、任务、习惯、偏好、洞察等），边代表实体间的关系（属于、导致、建议、相关等）。

星图的独特之处在于其**"以用户为中心"的视角**：用户本人是图的中心节点（root），所有其他节点都通过某种路径与用户关联。这种设计强化了"这是我的星图"的归属感，也使信息的组织和浏览符合用户的自我认知结构。

##### 2.2.1.2 分层信息架构（L1 概览/L2 类别/L3 详情/L4 洞察）

星图采用**严格的分层信息架构**，解决复杂信息的可理解性和可导航性问题。四个层级的设计遵循"从抽象到具体、从静态到动态"的认知规律：

| 层级     | 名称             | 内容范围                      | 节点数量   | 视觉特征                     | 交互方式               |
|:------ |:-------------- |:------------------------- |:------ |:------------------------ |:------------------ |
| **L1** | 概览（Overview）   | 用户核心身份、当前首要目标、整体状态指标      | 3-7个   | 最大节点，深海蓝填充，中心辐射布局        | 默认展示，点击进入L2        |
| **L2** | 类别（Categories） | 人生领域分类（学业/职业/健康/关系/兴趣/成长） | 6-12个  | 大节点，天际蓝填充，环形/网格布局        | 点击展开该类别下的L3        |
| **L3** | 详情（Details）    | 具体的目标、任务、习惯、资源、事件         | 20-50个 | 中等节点，浅蓝填充，力导向图布局         | 点击节点查看详情或展开L4      |
| **L4** | 洞察（Insights）   | AI生成的深度分析、关联发现、趋势预测、建议    | 动态生成   | 小节点，特殊形状（六边形/星形），发光/脉冲效果 | 以卡片/报告形式呈现，支持收藏和分享 |

分层架构的实现依赖节点的`level`属性和`parent_node_id`外键。L1节点无父节点（parent_node_id为null），L2-L4节点逐级指向父节点，形成树状的主干结构。同时，通过`star_edges`表的跨层级边，实现非树状的复杂关联（如一个习惯支持多个目标，一个洞察关联多个类别）。

用户的浏览路径通常是：从L1获得整体感知→选择感兴趣的类别进入L2→探索该领域的具体内容L3→发现AI生成的深度洞察L4。每一层的进入和退出都有平滑的动画过渡，保持空间方位的连续性，降低认知迷失。

##### 2.2.1.3 动态节点生成（任务/目标/习惯/偏好/长期目标）

星图的节点不是静态预定义的，而是**基于用户数据和AI分析动态生成**的。节点类型体系设计如下：

| 节点类型                 | 定义                         | 典型属性                    | 视觉标识                | 生成触发           |
|:-------------------- |:-------------------------- |:----------------------- |:------------------- |:-------------- |
| **目标节点（goal）**       | 用户明确设定的目标，包含描述、截止日期、进度、状态等 | 描述、截止日期、进度、状态、成功标准      | 六边形，深海蓝填充           | 用户明确声明、目标细化分解  |
| **任务节点（task）**       | 为实现目标而执行的具体行动              | 执行状态、时间安排、优先级、关联资源      | 矩形，天际蓝填充            | 目标分解、计划生成、对话提取 |
| **习惯节点（habit）**      | 用户重复性的行为模式                 | 频率、稳定性评分、触发条件、与目标的关联度   | 圆形，浅蓝填充，虚线边框        | 显式声明、AI行为模式推断  |
| **偏好节点（preference）** | 用户的稳定偏好                    | 学习风格、沟通语气、内容类型、活跃时段     | 菱形，淡紫填充             | 选择行为统计、直接询问确认  |
| **资源节点（resource）**   | 用户关注或AI推荐的书籍、文章、视频、工具、人脉等  | 类型、状态（已读/在用/想读）、关联目标/任务 | 文档图标，灰色填充           | 主动添加、AI推荐、计划关联 |
| **洞察节点（insight）**    | AI基于数据分析生成的深度发现            | 置信度、生成时间、数据来源、证据支撑      | 六边形/星形，金色/橙色填充，脉冲动画 | 定期AI分析、模式识别触发  |
| **事件节点（event）**      | 用户生活中的重要事件                 | 时间、类型、情绪标签、影响评估         | 旗帜形，红色填充            | 日历同步、对话提及、主动记录 |

节点的视觉编码：不同类型使用不同的形状，颜色按类别区分（学业-蓝、职业-绿、健康-橙、关系-粉、兴趣-紫、成长-青），大小反映重要性或紧急程度。节点生成后进入"候选"状态，经用户确认或一定时间无否定后转为"正式"状态，确保用户对星图内容的掌控感。

##### 2.2.1.4 AI 建议的下一步行动关联

星图不仅是信息的展示，更是**行动的触发器**。每个节点（尤其是目标、任务、洞察节点）都关联AI建议的下一步行动（`ai_suggestion`字段），这些建议基于节点的属性、关联节点的状态、以及相似用户的成功案例生成。

建议的生成策略包括：目标节点的"下一步"可能是分解子目标、设定里程碑、或启动相关任务；任务节点的"下一步"可能是开始执行、调整时间、寻求帮助、或标记完成；洞察节点的"下一步"可能是深入探索、设定相关目标、或调整现有计划。建议以可点击的按钮形式嵌入节点详情面板，一键触发对应的操作（跳转到深潜对话、创建计划项、添加到日历等）。

建议的**个性化程度**是深觅的核心竞争力。系统综合考虑用户的执行历史（过去对类似建议的采纳率）、当前状态（精力水平、时间可用性）、以及偏好设置（冒险型/稳健型、快速行动/深思熟虑），调整建议的内容和呈现方式。对于高采纳率的用户，建议更直接、更具挑战性；对于需要更多信息的用户，建议附带解释和案例。

#### 2.2.2 前端界面设计

##### 2.2.2.1 星图主画布（AntV G6 力导向图）

星图主画布是观己模块的视觉核心，采用**力导向图（Force-directed Graph）**布局算法，模拟物理系统的引力和斥力，使节点自动形成清晰可辨的聚类结构。

技术选型经过深入调研：ECharts 5生态成熟、文档完善、与Vue集成便捷；**AntV G6**在图可视化领域更专业，支持更复杂的交互和自定义，内置丰富的布局算法，性能优化深入（支持GPGPU加速和Web Worker布局计算）。深觅最终选择**AntV G6**，因其在知识图谱场景的专业优势，以及DB-GPT等生产级项目的验证。

画布配置参数经过调优以适应星图特性：布局采用force2算法，设置中心引力防止节点飘散，调整边长度使同层节点间距适中；节点大小映射重要性评分，边宽度映射关系强度；启用局部渲染优化，视口外节点不绘制；动画采用渐进式布局，新节点从父节点位置生长出现，增强因果关系感知。

画布的交互功能包括：缩放（鼠标滚轮/双指捏合，范围0.1x-5x）、平移（拖拽/双指滑动）、节点选中（单击高亮，关联边高亮）、节点展开（双击或点击展开按钮，加载子节点）、聚焦（点击"聚焦"按钮，以选中节点为中心重新布局）。画布的背景采用极浅灰蓝色（#F0F4F8），节点和边使用品牌蓝色系，确保视觉一致性。

性能优化策略：节点数量超过100时启用画布级别裁剪（只渲染视口内的节点），超过500时启用节点聚合（将密集区域的节点合并为超级节点，点击后展开）。动画效果用于布局过渡（力导向图的迭代过程可视化）和交互反馈（选中、hover的状态变化），时长控制在300ms以内，避免视觉疲劳。

##### 2.2.2.2 节点层级视觉区分（颜色/大小/形状）

节点的视觉编码系统确保用户能够快速识别节点的类型和重要性，无需阅读文字即可获取关键信息：

| 编码维度    | 规则                            | 具体实现                                                                                 |
|:------- |:----------------------------- |:------------------------------------------------------------------------------------ |
| **颜色**  | 按节点所属的人生领域（category）着色        | 学业（#4A90D9）、职业（#50C878）、健康（#FF8C42）、关系（#FF6B9D）、兴趣（#9B59B6）、成长（#1ABC9C）、未分类（#95A5A6） |
| **饱和度** | 映射层级（L1最高、L4最低）               | L1: 100%，L2: 85%，L3: 70%，L4: 55%                                                     |
| **大小**  | 映射重要性评分（importance_score，0-1） | 基础大小40px，最大放大至100px，公式：size = 40 + 60 * importance_score                             |
| **形状**  | 区分节点类型                        | 目标-六边形、任务-矩形、习惯-圆形、偏好-菱形、资源-文档图标、洞察-星形、事件-旗帜形                                        |
| **边框**  | 标识状态                          | 选中：2px深海蓝边框；hover：1px浅蓝边框；新节点：脉冲动画                                                   |

形状内部可叠加状态指示器：进行中-旋转进度环、已完成-对勾、逾期-感叹号、有未读建议-小圆点徽章。

##### 2.2.2.3 点击展开/折叠交互

点击展开/折叠是星图最核心的导航交互，实现分层信息架构的渐进披露：

| 交互动作     | 触发方式                         | 动画效果                                          | 状态更新                                |
|:-------- |:---------------------------- |:--------------------------------------------- |:----------------------------------- |
| **展开**   | 单击节点上的展开按钮、双击节点、详情面板中的"查看子项" | 子节点从父节点位置辐射出现（弹性动画，400ms，ease-out-back），边逐渐绘制 | 节点标记expanded=true，子节点加入图实例          |
| **折叠**   | 单击已展开节点的折叠按钮                 | 子节点以反向动画收缩回父节点，关联边消失                          | 节点标记expanded=false，子节点从图实例移除但保留位置记忆 |
| **智能折叠** | 长按折叠菜单选择"折叠全部后代"             | 批量移除所有后代节点，平滑过渡到新布局                           | 递归更新所有后代节点的展开状态                     |

展开动画采用物理弹簧模型，节点运动带有自然的加速和减速，避免机械感。展开过程中，非相关节点适度淡化（透明度降至30%），突出当前关注区域。面包屑导航显示当前的路径层级（如"我的星图 > 学业 > 考研目标 > 英语复习"），点击任意层级快速回退，解决深层浏览的迷失问题。

##### 2.2.2.4 侧边详情面板（选中节点信息）

侧边详情面板是节点信息的深度展示区域，采用**滑入式面板设计**（移动端从底部滑出，桌面端从右侧滑出），宽度为视口宽度的40%（桌面端最小360px，最大480px）。

面板内容根据节点类型动态生成，通用结构包括：

| 区域        | 内容                                     | 交互功能                                                       |
|:--------- |:-------------------------------------- |:---------------------------------------------------------- |
| **头部区域**  | 节点图标+名称+类型标签+操作按钮（编辑、删除、分享、收藏）         | 名称支持inline编辑，点击后直接修改                                       |

#### 2.2.3 数据结构

##### 2.2.3.1 节点类型定义（root/category/task/action/insight）

星图节点的数据模型设计需要支持灵活的类型扩展和丰富的属性存储：

```typescript
interface StarNode {
  node_id: string;           // UUID，全局唯一标识
  user_id: string;           // 外键，关联用户
  node_type: 'root' | 'category' | 'task' | 'action' | 
             'insight' | 'habit' | 'preference' | 'goal' | 'emotion';
  title: string;             // 节点显示名称，≤100字符
  description?: string;      // 详细描述，支持Markdown，≤2000字符
  category?: string;         // 所属人生领域：study/career/health/relationship/interest/growth
  level: 1 | 2 | 3 | 4;      // 层级
  parent_node_id?: string;   // 父节点ID，自引用外键，root为null

  // 类型特定字段，存储于metadata或独立字段
  goal_fields?: {
    deadline?: string;       // ISO 8601日期
    criteria?: string;       // 成功标准
    progress?: number;       // 0-100
  };
  task_fields?: {
    status: 'pending' | 'in_progress' | 'completed' | 'cancelled';
    priority: 'low' | 'medium' | 'high' | 'urgent';
    due_date?: string;
    estimated_minutes?: number;
  };
  habit_fields?: {
    frequency: 'daily' | 'weekly' | 'custom';
    streak_count: number;
    trigger_time?: string;
    best_streak: number;
  };
  insight_fields?: {
    confidence_score: number;  // 0-1
    evidence_count: number;
    generated_from: 'pattern_analysis' | 'trend_prediction' | 'anomaly_detection';
  };

  // AI生成与关联
  ai_suggestion?: {
    content: string;
    generated_at: string;
    confidence: number;
    accepted_at?: string;
  };
  related_memory_ids: string[];  // 关联记忆ID列表
  related_node_ids: string[];    // 关联节点ID列表

  // 元数据
  metadata: Record<string, any>; // 扩展字段
  importance_score: number;      // 0-1，计算属性
  access_count: number;          // 访问计数
  last_accessed_at?: string;     // ISO 8601时间戳
  created_at: string;
  updated_at: string;
}
```

##### 2.2.3.2 边关系类型（belongs_to/leads_to/suggests/relates_to）

边数据模型表达节点间的语义关系：

```typescript
interface StarEdge {
  edge_id: string;           // UUID
  from_node_id: string;      // 源节点
  to_node_id: string;        // 目标节点
  relation_type: 'belongs_to' |    // 归属：子节点→父节点
                 'leads_to' |      // 因果/顺序：前因→后果
                 'suggests' |      // 建议：AI推荐的关联
                 'relates_to' |    // 相关：双向弱关系
                 'enables' |       // 使能：资源/能力→目标
                 'conflicts_with'; // 冲突：互斥节点，需用户关注

  weight: number;              // 0-1，关系强度
  is_bidirectional: boolean;   // 是否双向关系
  is_inferred: boolean;        // 是否AI推断（vs用户明确建立）
  evidence_text?: string;      // 关系证据（如对话原文摘要）
  confidence_score: number;    // 推断置信度

  created_at: string;
  updated_at?: string;         // 关系属性更新时
}
```

**唯一约束**：(from_node_id, to_node_id, relation_type) 组合唯一，避免重复边。对于双向关系（如relates_to），存储两条反向边或标记is_bidirectional=true。

##### 2.2.3.3 层次编码规则（level + parent_node_id）

层级判定算法：根节点（无parent_node_id）自动为L1；子节点的level = parent.level + 1，最大4；超出4层的节点强制提升为洞察节点或拆分为独立子图。

**路径字段（path）** 作为物化路径优化查询：存储从root到当前节点的完整node_id序列，格式为`/root_id/category_id/task_id`。这一冗余设计支持高效的前缀匹配查询，如"获取某类别下的所有后代"可通过`WHERE path LIKE '/root_id/category_id/%'`实现。

**循环检测**：父子关系变更前检测是否形成环，防止无限递归。检测算法采用DFS，时间复杂度O(N)，N为节点数。检测到环时拒绝操作并提示用户。

**动态层级调整**：系统定期运行层级优化算法，分析节点的连接度、中心性、访问频率，识别"过度下沉"的重要节点和"虚高"的次要节点，提出层级调整建议，以"星图整理报告"形式推送用户确认。

### 2.3 追觅（计划制定与自主执行）

#### 2.3.1 功能定义

##### 2.3.1.1 基于星图与记忆的智能计划生成

追觅模块的计划生成功能是连接"观己"认知与"行动"落地的关键桥梁。计划生成基于多维度输入：**星图节点**（用户选定的目标及其关联子图）、**记忆上下文**（相关的能力评估、时间偏好、历史经验）、**约束条件**（用户明确的时间、资源、优先级限制），输出结构化的可执行计划。

生成流程采用 **"逆向规划+正向调度"的两阶段策略**：

| 阶段       | 核心任务                    | 技术实现                             | 输出                       |
|:-------- |:----------------------- |:-------------------------------- |:------------------------ |
| **逆向规划** | 从最终目标出发，递归拆解为子目标与里程碑    | 基于星图结构的图遍历，识别关键路径和依赖关系           | 目标树（Goal Tree），包含层次化的子目标 |
| **资源匹配** | 为每个子目标关联知识库中的学习资源、工具、人脉 | RAG检索相似案例的成功资源，结合用户偏好过滤          | 资源映射表，每个节点关联推荐资源列表       |
| **时间排程** | 考虑用户历史效率模式，分配时间节点       | 分析preferred_time、历史任务完成时长，避免过度乐观 | 时间线草案，含里程碑日期和缓冲期         |
| **风险预判** | 识别潜在障碍，生成预案             | 基于相似用户的失败模式，提示常见风险点              | 风险登记册，含预警信号和应对策略         |
| **整合优化** | 综合多目标约束，生成平衡方案          | 多目标优化算法，最大化完成概率、最小化资源冲突          | 完整计划文档，支持用户调整和确认         |

计划输出采用**OKR-style结构**：Objective（目标陈述，激发动力）、Key Results（3-5个可衡量的关键结果）、Initiatives（具体行动项，关联星图节点）、Resources（推荐资源，一键添加）、Timeline（可视化时间线，支持拖拽调整）、Risks & Mitigations（风险与预案）。

##### 2.3.1.2 IronClaw 自主任务执行

IronClaw是追觅实现AI从"建议"到"行动"跨越的关键组件，承担将AI生成的计划转化为具体动作的职责。在深觅的应用场景中，IronClaw负责任务分解、工具选择、执行监控、结果汇总，最终将执行日志和结果回调给主后端。

| 任务类型     | 典型场景                    | 执行流程                           | 人机协作模式        |
|:-------- |:----------------------- |:------------------------------ |:------------- |
| **信息检索** | 搜索考研院校报录比、查询课程评价、监控话题动态 | 分解查询关键词→多源并行搜索→结果去重排序→生成摘要报告   | 自动执行，结果推送用户确认 |
| **内容分析** | 分析学校录取数据、对比课程优劣、生成可视化报告 | 获取原始数据→清洗转换→统计分析→图表生成→洞察提炼     | 自动执行，关键发现高亮提示 |
| **工具操作** | 创建日历提醒、发送邮件、同步待办事项      | 解析操作意图→调用OAuth API→确认执行结果→处理异常 | 敏感操作需用户预授权    |
| **交易执行** | 预订课程、购买书籍、预约咨询服务        | 查询选项→比价推荐→用户确认→完成支付→订单追踪       | **强制人工确认节点**  |

IronClaw与深觅其他组件的集成遵循 **"AI决策、系统执行、结果反馈"的闭环模式**。异步架构确保用户界面的响应性：长时任务后台持续执行，完成后通过推送通知告知用户。执行日志完整记录于`ai_actions`表，支持事后审计与模型微调。

##### 2.3.1.3 当前任务辅助（学校分析/书籍推荐等）

任务辅助是追觅的"即时支持"功能，针对用户当前进行中的具体任务提供情境化帮助。典型场景覆盖：

| 场景类别     | 具体功能                | 输入                       | 输出                      | 集成点                |
|:-------- |:------------------- |:------------------------ |:----------------------- |:------------------ |
| **教育规划** | 学校/项目分析、申请策略、备考规划   | 用户背景、目标领域、约束条件（地域、预算、时间） | 对比报告、申请时间线、资源清单         | 星图"学业"节点、计划"申请准备"项 |
| **职业发展** | 技能路径规划、简历优化、面试准备    | 当前技能评估、目标职位、行业趋势         | 学习路线图、简历修改建议、模拟面试       | 星图"职业"节点、技能节点关联    |
| **学习支持** | 书籍/课程推荐、学习进度诊断、难点突破 | 学习目标、当前水平、偏好风格、可用时间      | 个性化学习方案、资源推荐列表、进度追踪     | 星图"学习"节点、任务执行记录    |
| **健康管理** | 运动计划、饮食建议、睡眠优化      | 健康目标、当前习惯、身体数据（可选同步）     | 可执行的健康计划、提醒设置、进度反馈      | 星图"健康"节点、习惯追踪数据    |
| **情感支持** | 焦虑缓解、压力管理、关系建议      | 情绪状态描述、触发情境、已尝试方法        | 即时技巧（呼吸/冥想）、专业资源转介、长期策略 | 星图"情绪"节点、情感记忆关联    |

每个场景有预置的**工具链和知识库**，通过RAG增强的提示工程实现专业级支持。例如，学校分析场景集成排名数据、录取统计、校友评价、课程设置等多源信息，生成结构化的决策支持报告。

##### 2.3.1.4 情感支持辅助（焦虑缓解/情绪识别）

情感支持是coaching关系的核心维度，深觅通过多模态信号识别用户情绪状态，提供分层级的支持响应：

| 识别层级     | 信号来源               | 识别方法          | 响应策略               |
|:-------- |:------------------ |:------------- |:------------------ |
| **显式表达** | 用户直接描述情绪（"我很焦虑"）   | 关键词匹配+情感词典    | 立即响应，确认情绪，提供针对性支持  |
| **隐式线索** | 语言模式变化（语速、句式、表情符号） | 文本特征分析+历史基线对比 | 温和探询，确认感知，避免标签化    |
| **行为模式** | 使用频率、时段、功能选择的变化    | 时间序列分析+异常检测   | 主动关怀，推送"最近如何"式对话邀请 |
| **生理信号** | 可选同步的可穿戴设备数据       | 心率变异性、睡眠质量等指标 | 整合分析，提示身心关联，建议专业咨询 |

支持策略库包括：**认知重构**（挑战灾难化思维，提供替代视角）、**行为激活**（建议具体小行动，打破反刍循环）、**正念引导**（呼吸练习、身体扫描，即时调节）、**社会连接**（提醒支持资源，降低孤立感）、**意义探索**（ reconnect with values，激发内在动力）。所有策略均标注证据等级和适用边界，AI明确声明辅助角色，检测到严重困扰时谨慎转介专业资源。

##### 2.3.1.5 主动信息推送（兴趣预测）

主动推送是深觅从"响应式"向"预见式"进化的关键，基于用户画像和行为模式，预测可能感兴趣或需要的信息：

| 触发类型     | 触发条件                     | 内容类型                    | 示例                       |
|:-------- |:------------------------ |:----------------------- |:------------------------ |
| **时间触发** | 用户设定的提醒、周期性总结            | 进度回顾、里程碑预警、习惯 streak 状态 | "本周已完成3次运动，距离目标还差1次"     |
| **事件触发** | 星图节点状态变化、计划截止日期临近、相关新闻发生 | 行动建议、资源更新、时机提醒          | "你关注的Python课程本周限时优惠"     |
| **模式触发** | 检测到兴趣迁移、效率下降、社交孤立等       | 洞察分享、调整建议、关怀邀请          | "注意到你这周晚间学习效率下降，试试调整时段？" |
| **外部触发** | 行业动态、季节变化、社会事件           | 趋势解读、机会提示、风险预警          | "AI行业招聘季开始，你的技能储备如何？"    |

推送渠道与频率控制：应用内通知（优先级最高，即时）、邮件摘要（每日/每周，非紧急汇总）、移动端推送（紧急或高价值内容）。用户可设置**个性化控制**：推送频率（少/中/多）、内容偏好（重资源/重洞察/重行动）、免打扰时段、以及一键暂停所有推送的"专注模式"。

#### 2.3.2 前端界面设计

##### 2.3.2.1 计划看板（待办/进行中/已完成）

计划看板采用**Kanban风格的三列布局**（桌面端）或**三标签页**（移动端），直观呈现计划项的状态流转：

| 列/标签                 | 内容       | 视觉特征          | 交互功能                   |
|:-------------------- |:-------- |:------------- |:---------------------- |
| **待办（To Do）**        | 尚未开始的计划项 | 灰色边框，无填充进度    | 拖拽至"进行中"、点击展开详情、批量选择   |
| **进行中（In Progress）** | 已启动的计划项  | 天际蓝边框，部分填充进度条 | 更新进度、标记阻塞、触发IronClaw执行 |
| **已完成（Done）**        | 已完成的计划项  | 绿色填充，对勾图标，可折叠 | 查看总结、归档、重新激活           |

每张计划卡片包含：标题、进度条（百分比或子任务完成数）、截止日期标签（正常-灰色、3天内-橙色、逾期-红色）、关联星图节点图标（点击跳转）、快捷操作按钮（开始/暂停/完成/更多）。卡片支持拖拽变更状态、点击展开详情、长按快速操作。

**批量操作与筛选**：框选多卡片，批量设置截止日期、分配给IronClaw、归档。过滤器支持按目标领域、优先级、时间范围、执行模式（手动/自动）筛选，帮助用户聚焦当前关注。

##### 2.3.2.2 计划详情页（步骤分解/资源关联）

计划详情页采用**两栏布局**：左侧步骤列表（可折叠的分组），右侧资源与笔记面板。

| 区域        | 内容                  | 交互特性                         |
|:--------- |:------------------- |:---------------------------- |
| **步骤列表**  | 层级化的任务分解，支持嵌套2层     | 复选框完成标记、拖拽排序、展开/折叠子步骤、显示依赖关系 |
| **时间线视图** | 甘特图风格的可视化           | 拖拽调整起止时间、自动检测冲突、显示里程碑        |
| **资源面板**  | 关联的书籍、文章、工具、人脉      | 内嵌预览、一键打开、标记完成状态、添加个人笔记      |
| **笔记区域**  | 支持Markdown，@引用步骤或资源 | #添加标签，@提及创建关联，实时同步           |
| **协作功能**  | 分享计划（只读/可编辑链接）、评论讨论 | @深觅获取AI建议，@其他用户邀请协作          |

##### 2.3.3.1 IronClaw 任务分解接口

深觅与IronClaw的接口协议采用**声明式任务描述**，便于AI生成和人类理解：

```json
{
  "task_id": "uuid",
  "task_type": "search|generate|analyze|operate|notify",
  "description": "自然语言任务描述，供AI理解和用户确认",
  "parameters": {
    "query": "Python online courses 2024",
    "constraints": {"max_results": 5, "price_range": "free-$50", "format": "comparison_table"},
    "output_format": "structured_json|markdown|html|plain_text"
  },
  "context": {
    "user_id": "uuid",
    "related_node_ids": ["n1", "n2"],
    "deadline": "2026-06-01",
    "priority": "high|medium|low"
  },
  "execution_policy": {
    "mode": "auto|confirm_before|confirm_each_step",
    "timeout_seconds": 300,
    "retry_policy": {"max_attempts": 3, "backoff": "exponential"},
    "fallback_action": "notify_user|escalate_human|skip_and_continue"
  },
  "callback_url": "https://shenmi.ai/webhook/ironclaw"
}
```

任务分解器将高层计划目标，输出可执行的动作序列，采用**层次化任务网络（HTN）**方法：识别目标类型→匹配预定义任务模板→实例化变量→检查依赖关系→确定执行顺序。

##### 2.3.3.2 执行状态回调机制

IronClaw通过**Webhook异步回调**返回状态更新，支持长轮询和SSE作为降级方案：

```json
{
  "task_id": "uuid",
  "action_id": "uuid",
  "status": "queued|running|completed|failed|needs_confirmation|cancelled",
  "progress": {"current_step": 3, "total_steps": 5, "percentage": 0.6},
  "current_activity": "正在比较课程评价...",
  "intermediate_results": [...],  // 多步骤任务的中间产出
  "result": {
    "summary": "找到5个符合条件的课程",
    "detail": {...},  // 结构化完整结果
    "artifacts": [
      {"type": "table", "content": "...", "format": "markdown"},
      {"type": "chart", "data": [...], "library": "echarts"}
    ]
  },
  "confirmation_request": {  // 若status=needs_confirmation
    "prompt": "需要我为你注册试听课吗？",
    "options": [
      {"id": "yes", "label": "是的，继续"},
      {"id": "more", "label": "提供更多选项"},
      {"id": "no", "label": "暂时不需要"}
    ],
    "timeout_seconds": 3600
  },
  "error": {  // 若status=failed
    "code": "EXTERNAL_API_ERROR",
    "message": "课程平台API返回503错误",
    "retryable": true,
    "suggested_fallback": "手动搜索或更换数据源"
  },
  "logs": [...],  // 完整执行日志
  "timestamp": "2026-03-11T14:30:00Z",
  "next_expected_update": "2026-03-11T14:32:00Z"
}
```

状态机：`pending → running → (completed | failed | needs_confirmation) → (confirmed | cancelled)`。超时处理：运行中任务10分钟无更新自动标记为失败，通知用户并记录诊断信息。

##### 2.3.3.3 人机协作确认节点

确认节点的设计**平衡效率与安全**，三级授权策略：

| 级别       | 条件                       | 行为                    | 示例               |
|:-------- |:------------------------ |:--------------------- |:---------------- |
| **自动通过** | 任务类型在白名单、无敏感参数、历史成功率>90% | 立即执行，结果异步通知           | 常规信息搜索、内容生成      |
| **智能确认** | AI生成操作建议，用户一键确认或修改后确认    | 展示"AI将要做什么"、预期结果、可能风险 | 日历创建、邮件发送、资源预订   |
| **强制确认** | 涉及支付、隐私数据、不可逆操作、首次使用的工具  | 明确请求授权，倒计时无响应则暂停      | 课程购买、个人信息提交、账户操作 |

确认界面设计：**清晰展示**（步骤列表、基于什么、可能结果）、**便捷操作**（一键确认/修改/拒绝/稍后）、**完整审计**（记录决策时间和用户身份）。对于需要人工确认的任务，IronClaw执行前暂停，通过推送通知+应用内提醒请求用户授权，形成人机协作的安全边界。

## 3. 前端设计详规

### 3.1 技术选型

#### 3.1.1 框架：Vue.js 3 + TypeScript

深觅前端采用 **Vue.js 3** 作为核心框架，配合 **TypeScript** 实现类型安全的开发体验。Vue 3的**组合式API（Composition API）** 提供了更灵活的逻辑复用能力，与深觅多模块（深潜/观己/追觅）的架构天然契合。`setup`语法和`<script setup>`的简洁性，使得组件代码更加清晰可维护。TypeScript的静态类型检查在AI应用开发中尤为重要——LLM返回的结构化数据、API响应格式、组件Props等均可获得类型安全，从根本上消除类型不一致导致的bug。

项目结构采用 **Feature-Based组织**：按功能模块（chat/star/explore）划分视图、组件、状态、API调用，而非技术类型（components/store/）。这种结构使得功能迭代更加聚焦，团队协作时冲突更少。性能优化方面：异步组件加载（星图ECharts动态导入）、虚拟滚动（长对话列表）、KeepAlive缓存（切换标签页保留状态）。开发体验：Vite快速HMR、ESLint + Prettier统一代码风格、Vitest单元测试 + Playwright E2E测试。

#### 3.1.2 UI 组件库：Ant Design X（AI 对话组件）

**Ant Design X** 是蚂蚁集团专为AI对话场景设计的React UI库，深觅借鉴其设计范式和技术架构，在Vue生态中寻找等效方案或自研实现。核心借鉴的**RICH设计范式**包括：

| 维度                   | 设计原则         | 深觅实现                       |
|:-------------------- |:------------ |:-------------------------- |
| **Intention（意图）**    | 意图识别过程可视化    | RAI Badge实时展示识别结果，支持用户确认修正 |
| **Role（角色）**         | 角色感知与切换      | 不同Coach角色对应不同头像、语气、界面主题    |
| **Conversation（对话）** | 对话流的自然呈现     | Bubble组件变体、思考过程展开、混合内容渲染   |
| **Hybrid UI（混合界面）**  | AI与人工协作的界面模式 | 建议操作嵌入对话、快捷指令、人机确认节点       |

关键组件实现：ChatBubble（对话气泡，支持变体与加载态）、Sender（输入框，支持语音/附件/快捷指令）、Conversations（会话列表管理）、Prompts（提示词模板）、ThoughtChain（思维链展示，用于复杂推理过程可视化）。主题定制通过CSS变量覆盖，将Ant Design X的默认蓝色映射为深觅品牌色。

#### 3.1.3 图表库：ECharts 5 + AntV G6（星图可视化）

图表库采用**双轨策略**：**ECharts 5** 负责常规数据可视化（计划进度、情绪趋势、学习曲线等），**AntV G6** 专责星图的关系图可视化。

| 维度       | ECharts 5         | AntV G6             |
|:-------- |:----------------- |:------------------- |
| **核心优势** | 生态成熟、图表类型丰富、配置简单  | 图算法专业、交互定制灵活、大规模性能优 |
| **适用场景** | 折线/柱状/雷达/热力图等统计图表 | 力导向图、知识图谱、复杂网络关系    |
| **性能表现** | 百万级数据点渲染          | 万级节点、十万级边流畅交互       |
| **深觅应用** | 计划进度、情绪趋势、效率分析    | 个人星图主画布、关系路径分析      |
| **扩展能力** | 插件生态丰富            | 自定义布局算法、GPGPU加速     |

ECharts 5的主题系统支持深觅品牌色的全局注入，保持视觉一致性。AntV G6的选型基于深度调研：G6在图可视化领域更专业，内置丰富的布局算法（force、circular、dagre等），支持GPGPU加速和Web Worker布局计算，性能足以支撑千级节点的流畅交互。DB-GPT万星开源项目的GraphRAG组件即采用AntV G6引擎实现知识图谱的酷炫渲染，验证了该技术方案在生产环境中的可靠性。

#### 3.1.4 状态管理：Pinia

**Pinia** 作为Vue官方推荐的状态管理方案，取代Vuex成为深觅的选择。其优势在于：完整的TypeScript支持（无需额外的类型封装）、更简洁的API设计（无mutations，直接状态修改）、模块化的Store组织、Devtools集成优化。

深觅的Store按领域划分：

- `userStore`：用户认证与画像、个人设置、隐私偏好
- `chatStore`：当前会话、历史记录、意图上下文、消息草稿
- `starStore`：星图数据（节点/边）、视图状态（缩放/中心/展开）、选中节点、加载状态
- `planStore`：计划列表、当前计划、执行状态、IronClaw任务追踪
- `uiStore`：主题模式、断点状态、全局加载、通知队列

跨Store通信通过组合式函数实现，避免循环依赖。持久化策略：用户敏感设置加密存储于localStorage，星图视图状态sessionStorage，大体积数据（历史消息）按需加载不持久化。

#### 3.1.5 响应式方案：CSS Grid + Flexbox + 断点适配

布局系统采用现代CSS技术栈：

- **CSS Grid**：页面级二维布局（星图画布与面板的并排/堆叠切换）
- **Flexbox**：组件级一维布局（导航、列表、表单）
- **容器查询（Container Queries）**：组件级的响应式（同一组件在不同父容器中的自适应）
- **媒体查询（Media Queries）**：页面级的断点切换

**8px网格系统**：所有间距、尺寸均为8的倍数（8, 16, 24, 32, 48, 64），保证视觉节奏的一致性。**CSS变量（Custom Properties）** 集中管理设计令牌（颜色、间距、圆角、阴影），支持运行时主题切换（浅色/深色模式）。

## 6. 关键流程与状态机

### 6.1 对话处理流程（LangGraph）

```
[输入节点] → [意图识别] → [记忆检索] → [角色选择] → [回复生成] → [记忆更新] → [星图同步]
                ↓（置信度低）      ↓（无相关记忆）
           [澄清子流程]        [通用回复分支]
```

### 6.2 星图构建流程

| 阶段   | 触发条件              | 核心操作                        | 输出   |
|:---- |:----------------- |:--------------------------- |:---- |
| 初始构建 | 新用户首次使用、手动触发重建    | 从记忆和对话自动提取实体，识别关系，生成L1-L2结构 | 基础星图 |
| 增量更新 | 新对话产生、计划完成、用户主动添加 | 提取新实体，更新现有节点属性，调整关系权重       | 变更日志 |
| 层次聚合 | 定期（每周）AI分析任务      | 识别可聚合的低层节点，生成高层洞察，用户确认后提升层级 | 洞察报告 |
| 实时同步 | 对话中关键信息提取         | 异步触发节点创建或关系更新，前端增量刷新        | 更新推送 |

### 6.3 计划执行流程（IronClaw）

```
[计划生成] → [任务分解] → [动作排队] → [执行调度] → [工具调用] → [结果处理] → [用户通知]
                ↓（需确认）              ↓（失败）
           [确认请求] ←——————— [重试/降级] ←——————— [异常处理]
```

## 7. 性能与扩展考量

### 7.1 查询优化

| 场景     | 优化策略                                           | 预期效果              |
|:------ |:---------------------------------------------- |:----------------- |
| 星图节点查询 | 复合索引(user_id, level, status) + 物化路径path字段      | 延迟<50ms           |
| 向量搜索   | Chroma HNSW索引，M=16, efConstruction=200         | 召回率>95%, 延迟<100ms |
| 记忆检索   | 按importance_score + last_accessed_at排序，LRU缓存热点 | 命中率>80%           |

### 7.2 缓存策略

| 层级         | 策略           | TTL  |
|:---------- |:------------ |:---- |
| L1 进程内缓存   | 用户级星图数据、会话状态 | 60s  |
| L2 Redis缓存 | 星图查询结果、推荐列表  | 300s |
| L3 数据库     | 持久化存储，读写分离   | —    |

### 7.3 扩展路径

| 阶段                  | 触发条件                  | 迁移方案                        |
|:------------------- |:--------------------- |:--------------------------- |
| SQLite → PostgreSQL | 用户规模>10万，并发写入瓶颈       | SQLAlchemy方言切换，零业务代码改动      |
| 单机Chroma → 分布式      | 向量规模>1000万，查询延迟>500ms | 迁移至Milvus/Pinecone，API兼容层封装 |
| IronClaw插件生态        | 任务类型扩展需求              | 定义标准插件接口，WASM沙箱执行           |
